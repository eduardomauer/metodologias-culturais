# Governance Policy

This repository is the public technical documentation layer for MILK cultural methodologies, interoperability, licensing, accessibility and public-interest digital infrastructure.

It is not the private operational repository of the Atlas Vivo MILK, the OCSR, protected visual assets, internal research records or confidential strategy.

## Responsible entity

Association: **Associação MILK — Movimento de Intervenções e Linguagens Kulturais e Arte**  
Public authorial and technical layer: **Eduardo Mauer**  
Repository scope: public, reviewed technical and methodological documentation.

## Governance principles

1. **Public-readiness before publication.** Every file must be reviewed before public release.
2. **Human validation.** Automated tools may assist, but final validation is human.
3. **No regression.** Validated structure, authorship, licence and access level must not be weakened.
4. **Layer separation.** Public documentation must remain separate from protected methodology, assets and internal operations.
5. **Provenance.** Public files must state source, responsible entity, version, date, licence and access level.
6. **Institutional clarity.** Associação MILK, Atlas Vivo MILK, Eduardo Mauer and Nuno A. must remain distinct where their roles differ.

## Public release rule

A file is public-ready only when it has:

- clear title;
- responsible entity or author;
- version;
- date;
- source or provenance;
- licence or reuse condition;
- access level;
- validation status;
- no restricted operational material.

## Access levels

| Level | Meaning | Publication status |
|---|---|---|
| Public | Reviewed documentation prepared for citation or public reading | May be published |
| Protected | Methodological, visual or operational layer not intended for open release | Keep outside this repository |
| Internal | Controlled-access material requiring human review | Keep outside this repository |
| Pending review | Material awaiting provenance, licence or validation check | Do not publish yet |

## Change control

Substantive changes should be made on a separate branch. Each change should explain:

- what changed;
- why it changed;
- which files were affected;
- what remained intact;
- regression risk;
- validation method.

## Review authority

Final validation remains with the human maintainer or responsible institutional authority. Automated systems must not decide authorship, licence, public-readiness, legal sufficiency, ethical status or institutional release.
